## Context

当前 Hive 的 Agent 执行是单次通过模式：用户任务 → Main Agent 执行 → 返回结果。`RuntimeResult.steps` 包含完整的工具调用记录，但在映射为 `DispatchResult` 时被丢弃。

LLM 是文本预测器，生成"已完成"声明时不会自动对照实际工具调用记录。幻觉的根本原因：**文本生成与实际执行之间没有校验闭环**。

此前设计了 GAN discriminator 方案（双层验证 + 复杂度标签 + 关键词映射表），但过于复杂（24 tasks）且入口守卫（复杂度标签）恰恰在最易幻觉的"简单任务"上失效。从第一性原理推导出三防线方案，用最小改动解决根本问题。

## Goals / Non-Goals

**Goals:**
- Agent 声明完成前必须对照执行记录
- 零工具调用的 action task 被拦截并重试
- 有工具调用但可能部分完成的任务通过 steps 注入自省纠正
- 纯问答任务零额外开销
- 实现量最小（改 prompt + ~40 行代码）

**Non-Goals:**
- 不引入外部验证器或额外模型
- 不引入复杂度自评标签
- 不引入关键词映射表
- 不改变子 Agent 执行逻辑
- 不改变 LLMRuntime、ToolRegistry、Hook 系统

## Decisions

### D1: 三防线架构（预防 → 拦截 → 自省）

**选择**: 三层串行防线，逐层升级。

```
防线 1 (prompt): 执行协议约束 Agent 行为         → $0, 雄 LLM 调用
防线 2 (代码):   零工具调用拦截 + 重试           → $0, 零 LLM 调用
防线 3 (自省):   steps 注入 + Agent 自行确认     → 复用当前模型 1 次调用
```

**理由**: 从根因出发——幻觉本质是"声明完成时没看过执行记录"。防线 1 预防，防线 2 拦截最常见幻觉（零工具调用），防线 3 覆盖更微妙的幻觉（部分完成）。三层递进，成本递增，覆盖率递增。

**替代方案**: GAN discriminator（双层验证 + Haiku）——过于复杂，复杂度标签在简单任务上失效，关键词表覆盖面有限。

### D2: action task 判定——暴力模式

**选择**: 所有非纯问答任务一律视为 action task。纯问答通过启发式排除（任务不包含动作动词且长度较短）。

**理由**: 宁可多验证一次，不放过幻觉。误判纯问答为 action task 的代价仅是一次额外 LLM 调用；漏判 action task 的代价是幻觉未被发现。

**替代方案**: 动词启发式匹配——更精准但可能漏判，收益不值得增加的复杂度。

### D3: 零工具调用拦截——注入反馈重试 1 次

**选择**: 检测到 `tools.length === 0` 且为 action task 时，注入反馈消息（"你还没有调用任何工具来执行任务，请使用合适的工具实际执行"），重新调用 LLMRuntime 1 次。

**理由**: 最多 1 次重试。如果重试后仍然零工具调用，说明问题在 prompt 层面或 Agent 确实认为不需要工具，继续重试收益递减。返回结果但附带警告信息。

**替代方案**: 多轮重试——成本增加但收益递减，不如改善 prompt。

### D4: steps 注入自省——复用当前模型 1 次调用

**选择**: 当 `tools.length > 0` 时，将 steps 格式化为摘要注入对话历史，追加用户消息"请确认你是否已完成所有操作。如未完成，请继续执行。"，复用当前模型额外调用 1 次。

**理由**: Agent 面对具体执行证据时自我纠正率高，无需额外模型。1 次调用足够——如果看到 steps 后仍然声称完成，大概率是真的完成了。

**替代方案**: 使用 Haiku 模型验证——增加模型切换复杂度，且 Haiku 本身可能产生验证幻觉。

### D5: DispatchResult 保留 steps

**选择**: 在 RuntimeResult → DispatchResult 映射时保留 `steps` 字段作为可选字段。

**理由**: steps 是验证的事实依据，也是重试反馈的核心数据。可选字段不破坏现有调用方。

### D6: 三防线仅作用于正常模式

**选择**: 三防线仅在 `forceMode === undefined`（正常模式）时生效。`forceMode === 'explore' | 'plan'`（只读模式）不触发，因为只读模式本身不执行写操作，不存在"虚假完成"风险。

**理由**: 只读模式只使用 explore 工具集（file 只读、glob、grep、web），不涉及文件修改/命令执行等动作，幻觉风险极低且无需验证。

## Risks / Trade-offs

**[Risk] 自省调用增加延迟和成本** → 仅 action task 触发，复用当前模型 1 次额外调用。简单任务零额外开销。最坏情况（action task + 零工具拦截 + 自省）额外 2 次 LLM 调用，延迟可控在 10s 内。

**[Risk] action task 误判** → 纯问答被误判为 action task 时，防线 2/3 的检查会快速通过（Agent 回复不含动作声明，自省确认无需额外操作），仅增加 1 次额外 LLM 调用的成本。

**[Risk] Agent 自省后仍然幻觉** → 概率极低（面对具体 steps 证据时 LLM 自我纠正率高）。即使发生，用户收到的仍是 Agent 的回复，只是未经最终确认——与当前行为一致（当前完全没有验证），不会比现状更差。

**[Risk] 反馈消息导致对话历史膨胀** → 最多注入 2 条额外消息（防线 2 反馈 + 防线 3 自省），每条 < 500 字符。DynamicPromptBuilder 已有 token budget 管理。
